## Q: MVC 패턴은 무엇인가요?

MVC 패턴은 모델, 뷰, 컨트롤러로 이루어진 디자인 패턴입니다. 
- **Model:** 데이터와 비즈니스 로직 담당.
- **View:** 사용자에게 보이는 화면(UI) 담당.
- **Controller:** 사용자 입력을 받아 모델을 조작하고, 뷰를 선택하여 업데이트.

**애플리케이션의 구성 요소를 세 가지 역할로 구분하여 개발 프로세스에서 각각의 구성 요소에만 집중해서 개발할 수 있다는 점**과 **재사용성과 확장성이 용이**하다는 장점이 있고, 애플리케이션이 복잡해질수록 **모델과 뷰의 사이의 의존성이 강해져 유지보수가 어려워지는 단점**이 있습니다.

> Q: 모델과 뷰의 의존성이 강해진다고 하셨는데, 구체적으로 어떤 상황에서 그런 문제가 발생하나요?

앱이 커지면 하나의 뷰가 여러 개의 모델을 참조하거나, 반대로 하나의 모델 변화가 여러 에 영향을 주게 됩니다. 결국 컨트롤러가 너무 방대해져서 코드 파악이 힘들어지는 **Massive View Controller** 문제가 생깁니다.

> Q: MVC 패턴에서 비즈니스 로직은 정확히 어디에 위치해야 한다고 생각하시나요?

비즈니스 로직은 **모델**에 위치해야 합니다. 컨트롤러는 흐름을 제어할 뿐, 데이터의 가공이나 규칙은 Model이 담당해야 코드 재사용성이 높아집니다.

> Q: React는 MVC 패턴인가요? 아니면 다른 패턴인가요?

React는 UI 라이브러리이므로 뷰에 집중하지만, 전체적으로는 **단방향 데이터 흐름(Flux 아키텍처)**를 지향합니다. MVC의 복잡한 양방향 데이터 흐름을 해결하기 위해 나타난 구조입니다.

> Q: 서버 사이드(Spring, Express 등)에서의 MVC와 클라이언트 사이드(React, Vue 등)에서의 MVC는 어떤 차이가 있을까요?

서버 측 MVC는 주로 요청에 맞는 HTML을 렌더링해서 주는 방식이고, 클라이언트 측 MVC는 데이터만 받아와서 동적으로 브라우저 화면을 직접 그리는 데 중점을 둡니다.
## Q: MVP 패턴과 MVC 패턴은 어떤 차이가 있나요?

MVP 패턴은 MVC 패턴으로부터 파생되었으며 MVC에서 C에 해당하는 컨트롤러가 프레젠터로 교체된 패턴입니다. 뷰와 프레젠터는 일대일 관계이기 떄문에 MVC 패턴보다 강한 결합을 가진 디자인 패턴입니다.

뷰와 모델은 서로를 전혀 모르며, 오직 프레젠터를 통해서만 소통합니다. 뷰와 모델의 의존성이 완전히 분리되어 테스트가 용이해집니다. 그러나 뷰가 늘어날 수록 프레젠터도 비례해서 늘어나 코드양이 많아집니다.
## Q: MVVM 패턴과 MVC 패턴은 어떤 차이가 있나요?

MVVM 패턴은 MVC의 C에 해당하는 컨트롤러가 뷰모델로 바뀐 패턴이며 데이터 바인딩을 핵심으로 하는 디자인 패턴입니다.

뷰모델은 뷰를 직접 참조하지 않고, 상태 데이터만 유지하게 됩니다. 커맨드와 **데이터 바인딩을 통해 뷰와 뷰 모델 사이의 데이터를 자동으로 동기화합니다.** 

뷰와 뷰모델 사이의 양방향 데이터 바인딩을 통해 UI를 별도의 코드 수정 없이 재사용할 수 있고 독립적인 단위 테스트가 매우 쉬워집니다. 그러나 설계 복잡도가 높고, 데이터 바인딩 과정에서의 메모리 소비 및 디버깅의 어려움이 있습니다.

> Q: MVVM에서 **데이터 바인딩**은 어떻게 동작하나요?

주로 [옵저버 패턴](Observer.md)을 활용합니다. ViewModel의 데이터(상태)를 Observable한 객체로 만들고, View가 이를 구독하고 있다가 데이터가 변경될 때 자동으로 UI를 갱신하게 만듭니다.

> Q: 그럼 무조건 MVVM이 제일 좋은 건가요?

아닙니다. MVVM은 데이터 바인딩을 위한 **설계 복잡도가 높고**, 소규모 앱에서는 오히려 **오버헤드**가 될 수 있습니다. 또한 데이터 바인딩 에러는 디버깅이 어렵다는 단점도 있습니다. 따라서 프로젝트의 규모와 팀의 숙련도에 따라 적절한 패턴을 선택하는 것이 중요합니다.
